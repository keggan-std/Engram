## Audit Findings

### Confirmed Issues (from feedback)

| # | Issue | Severity | Root Cause |
|---|-------|----------|------------|
| F1 | Session start output is too large, even at `verbosity:"summary"` | Medium | Front-loads all context; no lazy-load pattern |
| F2 | No `observations` action — agents forced to misuse `record_decision` | Low–Med | Missing semantic type between "sticky note" and "architecture decision" |
| F3 | Params format error on complex string input | High | Missing `coerceStringArray()` on 4 Zod fields |

### Additional Issues (discovered during audit)

| # | Issue | Severity | Root Cause |
|---|-------|----------|------------|
| D1 | **Universal mode bypasses ALL Zod preprocessing** | Critical | `HandlerCapturer` passes raw params directly to handlers, skipping `z.preprocess()` transforms |
| D2 | **`dump` action "finding" type is not persisted** | Medium | Finding branch generates an ephemeral ID (`Date.now() % 1_000_000`) but writes nothing to the DB |
| D3 | **5 `z.array(z.number())` fields lack coercion** | Medium | Same serialization bug as string arrays, but for number arrays |

### Agent-experienced issues (from v1.11-dx-experience-log.md)

The auditing agent itself experienced all 3 user-reported issues during the audit session:
- **3× `"must be object"` errors** on `set_file_notes`, `record_decision`, `create_task` (~20% failure rate on complex calls)
- **~800 tokens consumed** by session start before any work began (28% of Engram overhead was waste)
- **Forced to use `record_decision`** for factual observations (no semantic alternative)
- **Token efficiency**: 28% of Engram call tokens were wasted on oversized session start + failed retries

## Implementation Plan

### Workstream 1: Session Verbosity Overhaul

**Goal:** Reduce session start output from ~800 tokens to ~200 tokens by default, with on-demand expansion.

#### 1.1 — Already have: `nano`, `minimal`, `summary`, `full`

Current token efficiency by tier:
- `nano` (~80 tokens): counts + agent_rules + tool_catalog — too sparse, no decisions/tasks
- `minimal` (~300 tokens): counts + 5 capped conventions — skips decisions and tasks entirely
- `summary` (~700–900 tokens): 5 decisions + 10 conventions + 5 tasks + 5 changes — the user's complaint
- `full` (1200+ tokens): everything — rarely used

#### 1.2 — Fix: Improve `summary` tier compression

**Changes to `src/tools/sessions.ts` (summary branch ~L370):**

1. **Truncate decisions to 80 chars** (currently 120):
   ```typescript
   // Before
   decision: truncate(d.decision, 120)
   // After
   decision: truncate(d.decision, 80)
   ```

2. **Cap changes to 3** (currently 5) and drop `timestamp` field:
   ```typescript
   // Before
   recent: recordedChanges.slice(0, 5).map(c => ({ file_path: c.file_path, change_type: c.change_type, description: truncate(c.description, 120), timestamp: c.timestamp }))
   // After
   recent: recordedChanges.slice(0, 3).map(c => ({ file_path: c.file_path, change_type: c.change_type, description: truncate(c.description, 80) }))
   ```

3. **Cap conventions to 6** (currently 10):
   ```typescript
   capConventions(6) // was capConventions(10)
   ```

4. **Cap decisions to 3** (currently 5):
   ```typescript
   activeDecisions.slice(0, 3) // was slice(0, 5)
   ```

5. **Drop `git_log` from summary** (already available via `what_changed`):
   ```typescript
   git_log: undefined // was gitLog ? truncate(gitLog, 500) : ""
   ```

6. **Add count summary object** so agents know what's available without loading:
   ```typescript
   counts: {
     decisions: activeDecisions.length,
     conventions: totalConventions,
     tasks: openTasks.length,
     changes: recordedChanges.length,
     files: repos.fileNotes.countAll()
   }
   ```

**Estimated savings:** ~300 tokens, bringing `summary` from ~800 to ~500 tokens.

#### 1.3 — Improve `minimal` tier to include decision/task counts

The `minimal` tier already has counts but no decision or task info at all. Add a single most-important decision and most-urgent task:

```typescript
top_decision: activeDecisions[0] ? { id: activeDecisions[0].id, decision: truncate(activeDecisions[0].decision, 80) } : undefined,
top_task: openTasks[0] ? { id: openTasks[0].id, title: openTasks[0].title, priority: openTasks[0].priority } : undefined,
```

#### 1.4 — Constants extraction

Move all cap values to `src/constants.ts` for easy tuning:

```typescript

export const SESSION_SUMMARY_MAX_CONVENTIONS = 6;
```

### Workstream 2: `record_observation` Action

**Goal:** Provide a semantically correct, lightweight action for factual observations that don't have the weight of architecture decisions.

#### 2.1 — New `observations` table (Migration V24)

```sql
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id INTEGER,
  timestamp TEXT NOT NULL,
  observation TEXT NOT NULL,
  context TEXT,           -- optional: file path, line number, module
  category TEXT DEFAULT 'general',  -- 'codebase', 'behavior', 'performance', 'dependency', 'general'
```

```typescript
export function coerceNumberArray() {
  return z.preprocess((v) => {
    if (typeof v === "string") {
      try { return JSON.parse(v); } catch { return v; }
    }
    return v;
  }, z.array(z.number().int()));
}
```

#### 3.4 — Universal mode Zod preprocessing (CRITICAL)

**Problem:** `HandlerCapturer` passes raw params to handlers, completely bypassing Zod `z.preprocess()` transforms. This means ALL `coerceStringArray()` and `coerceNumberArray()` fixes in 3.1–3.3 **don't work in universal mode**.

**Solution:** Add a shared coercion preprocessor that runs before dispatching in universal mode.

**File:** `src/modes/universal.ts`

```typescript
/**
 * Pre-process params to handle MCP client serialization quirks.
 * Necessary because HandlerCapturer bypasses Zod z.preprocess() transforms.
 */
function coerceParams(params: Record<string, unknown>): Record<string, unknown> {
  const result = { ...params };
  for (const [key, value] of Object.entries(result)) {
    if (typeof value === "string") {
      // Try to parse JSON-stringified arrays/objects
      if (value.startsWith("[") || value.startsWith("{")) {
        try { result[key] = JSON.parse(value); } catch { /* keep as string */ }
      }
    }
    // Recurse into arrays of objects
    if (Array.isArray(value)) {
      result[key] = value.map(item => {
        if (typeof item === "object" && item !== null) {
          return coerceParams(item as Record<string, unknown>);
        }
        return item;
      });
    }
  }
  return result;
}
```

Apply before dispatch:
```typescript
const params = coerceParams({ action: resolvedAction, ...rest });
```

**Risk note:** This is a broadly-sweeping coercion. We must ensure it doesn't break params that are intentionally JSON strings (e.g., `action_data` in scheduled events). The `startsWith("[")` / `startsWith("{")` heuristic is safe because MCP params that are intentionally strings won't start with `[` or `{` in normal usage.

#### 3.5 — Defensive parsing at consumption sites

As a belt-and-suspenders measure, add `safeParseArray()` helper for repo-layer consumption:

**File:** `src/utils.ts`

```typescript
/**
 * Defensively parse a value that should be a string[] or number[].
 * Handles pre-serialized JSON strings from MCP clients.
 */
export function safeParseJsonArray<T>(value: unknown): T[] | null {
  if (value == null) return null;
  if (Array.isArray(value)) return value as T[];
  if (typeof value === "string") {
    try {
      const parsed = JSON.parse(value);
      return Array.isArray(parsed) ? parsed : null;
    } catch { return null; }
  }
  return null;
}
```
