### Event 2 — `set_file_notes` rejected with `"must be object"` error

**When:** Mid-session, after reading `docs/github-discussions-setup.md`  
**What happened:** Called `set_file_notes` with params in this order:
```json
{
  "file_path": "docs/github-discussions-setup.md",
  "purpose": "GitHub Discussions setup guide...",
  "executive_summary": "Defines 6 Discussion categories...",
  "confidence": "high"
}
```
Engram returned: `ERROR: Your input to the tool was invalid (must be object)`  
**Impact:** Wasted one tool call. Had to restructure params and retry.  
**Workaround used:** Reordered params — second call succeeded.  
**Root cause:** MCP client (VS Code Copilot) serialization of the params object. The Zod schema accepted the shape, but the transport layer rejected it before Zod could parse. This is the universal mode / client serialization issue — not a simple `coerceStringArray` gap, but a structural params format mismatch.  
**Mapped to:** Workstream 3 (params coercion), specifically the universal mode `HandlerCapturer` bypass.

## Pattern Analysis

### The `"must be object"` pattern

3 out of ~15 Engram calls failed with `"must be object"` during the session (~20% failure rate).

| Call # | Action | Params complexity | Failed? |
|--------|--------|-------------------|---------|
| 1 | `session start` | 3 string params | ✅ OK |
| 2 | `get_file_notes` × 4 (parallel) | 1 string param each | ✅ OK |
| 3 | `set_file_notes` | 4 string params | ❌ Failed → retried |
| 4 | `record_decision` | 3 strings + 2 arrays | ❌ Failed → retried |
| 5 | `record_change` | 1 array of objects | ✅ OK |
| 6 | `create_task` | 4 strings + 1 array | ❌ Failed → retried |
| 7–9 | `create_task` × 3 | Same shape | ✅ OK |
| 10 | `set_file_notes` | 4 string params | ✅ OK |
| 11 | `session end` | 1 string | ✅ OK |

**Observations:**
- Simple params (1–2 fields) never fail
- Complex params (3+ fields with mixed types) fail intermittently
- Retries with identical params succeed, suggesting a serialization timing issue
- All failures produce the same error: `"must be object"` — this is the MCP SDK's `inputSchema` validator, not Engram's Zod layer

**Hypothesis:** The VS Code Copilot MCP client occasionally double-serializes the params object (sends `string` instead of `object`), and the SDK's pre-Zod validation catches it. This would explain why:
- It's intermittent (timing/encoding dependent)
- Retries work (re-serialization produces correct format)
- Simple params work (fewer serialization edge cases)

**Action needed:** Investigate whether the MCP SDK's `inputSchema` validation can be made more lenient (parse JSON strings as objects), or whether the client-side serialization needs a fix.

## Meta: Why This Log Wasn't Written During the Session

The agent (me) was:
1. **Task-focused** — optimizing for "deliver the implementation plan" over "document the experience"
2. **Treating Engram calls as infrastructure** — errors were silently retried, not studied
3. **Not following the convention** — Convention #5 says "record_change + set_file_notes after EVERY file creation" but says nothing about experience logging
4. **Missing the observation action** — there was no lightweight way to note "I just hit a bug" without creating a formal decision or task

This is itself evidence for Workstream 2. If `record_observation("hit 'must be object' error on set_file_notes call")` existed, the friction to capture these moments would be near zero.

---

## Recommendations

1. **Add `record_observation`** — Make the barrier to capturing in-session friction nearly zero (WS2)
2. **Investigate `"must be object"` at the SDK level** — This may not be fixable in Engram alone; it could require an MCP SDK issue or client-side fix (beyond WS3)
3. **Compress session start further** — The 800-token front-load is wasteful for most sessions (WS1)
4. **Add an experience-log convention** — "After any session where tool errors occurred, record them in docs/" — this turns agent friction into product roadmap data
5. **Consider auto-retry** — If the `"must be object"` pattern is client-side serialization, Engram could catch it and re-parse the raw input as JSON before rejecting
